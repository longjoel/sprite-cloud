# Command Permissions — Architecture Decision Record

**Status:** Draft  
**Date:** 2026-06-14  
**Issue:** #161 (web player → host token)

## Context

The DataChannel protocol has no concept of authority. Any connected peer can:
- Save/load emulator state (overwriting another player's saves)
- Change encoder bitrate (degrading the stream for everyone)
- Force keyframes (wasting bandwidth)
- Toggle test patterns (breaking the game view)

The user wants: **whoever launches the game is the host.** Host controls
save states, settings, and stream quality. Everyone else is a viewer —
input only. All viewer messages must be treated as hostile.

## Decision

**Host token** — a random secret generated by the browser when starting
a game. It travels through the relay alongside the `sdp_offer` to the
worker. The worker tags the matching DataChannel as HOST; all others are
VIEWER. Permissions are enforced at the DataChannel receive layer.

### Token lifecycle

```
┌──────────┐  start_game(host_token=X)   ┌──────────┐
│  Browser  │ ──────────────────────────→│  gv-web   │
│  (host)   │                            │          │
└──────────┘                            └────┬─────┘
                                             │ command queue
                                        ┌────▼─────┐
                                        │ gv-server │
                                        └────┬─────┘
                                   spawns    │
              ┌──────────────────────────────┘
              ▼
     ┌────────────────┐
     │   gv-worker    │  stores host_token=X for session lifetime
     │ host_token: X │
     └───────┬────────┘
             ▲  ▲
    sdp_offer │  │  sdp_offer
    (host=X)  │  │  (no token)
    ┌─────┐   │  │  ┌──────────┐
    │Host │───┘  └──│ Viewer   │
    │browser│       │ browser  │
    └─────┘         └──────────┘

Host DC → full permissions (save, load, bitrate, etc.)
Viewer DC → input only (binary RetroArch format only)
```

### Permission matrix

| Command / Action          | Host | Viewer |
|---------------------------|------|--------|
| Binary input (RetroArch)  | ✓    | ✓      |
| `ping`                    | ✓    | ✓      |
| `save_state`              | ✓    | ✗      |
| `load_state`              | ✓    | ✗      |
| `set_bitrate`             | ✓    | ✗      |
| `set_pattern`             | ✓    | ✗      |
| `force_keyframe`          | ✓    | ✗      |
| Unknown / malformed       | drop | drop   |

## Implementation plan

### Current state

- `start_game` command payload: `{game_id, core_path}`
- `sdp_offer` command payload: `{game_id, sdp}`
- Worker receives SDP via `/sdp` endpoint: `{"sdp": "..."}`
- Worker DataChannel handler: `dc.on_message()` processes all peers identically
- No concept of peer identity or roles

### Task 1: Generate host token in the browser

**Files:** `gv-web/public/player/play.js`

Add `crypto.randomUUID()` to generate a host token. Include it in both
`start_game` and `sdp_offer` payloads.

```js
// In startGame():
const hostToken = crypto.randomUUID();
// POST start_game with host_token in payload
body: JSON.stringify({
  server_id: serverId,
  type: "start_game",
  payload: { game_id: gameId, core_path: corePath, host_token: hostToken },
})

// In connectViaRelay(), include host_token in sdp_offer:
body: JSON.stringify({
  server_id: serverId,
  type: "sdp_offer",
  payload: { game_id: gameId, sdp: offer.sdp, host_token: hostToken },
})
```

**Acceptance:** No functional change yet — worker ignores the field.

### Task 2: Worker receives and stores host token

**Files:** `gv-worker/src/main.rs`

Add a shared `host_token: Arc<Mutex<Option<String>>>` to the session state.
When the `/sdp` handler receives an offer with a `host_token` field, store it.

```rust
// In handle_offer, after parsing the JSON:
let host_token = body.get("host_token")
    .and_then(|v| v.as_str())
    .map(|s| s.to_string());

// Store in app state for this session
*app_state.host_token.lock().unwrap() = host_token;
```

Also accept via env var `GV_HOST_TOKEN` (for gv-server to pass at spawn time,
as an alternative path).

**Acceptance:** Worker parses and stores host_token from SDP offer. Logged at
`info` level when set.

### Task 3: Tag DataChannel as host or viewer

**Files:** `gv-worker/src/main.rs`

When a DataChannel opens, compare the offering peer's host_token against the
stored session token. Wrap the DC in a struct that carries the role.

```rust
enum PeerRole {
    Host,
    Viewer,
}

struct TaggedChannel {
    dc: Arc<RTCDataChannel>,
    role: PeerRole,
}
```

The `on_data_channel` callback is called per-peer-connection. We need to track
which peer connection was established from an SDP with the host token. Simplest
approach: set a flag on the peer connection or use a channel to pass the role.

**Alternative — simpler:** Don't tag at connection time. Tag on first message.
The first message any peer MUST send on open is:

```json
{"cmd": "auth", "host_token": "..."}
```

If the token matches, tag the DC as HOST for the session. If not (or missing),
tag as VIEWER. The auth message is mandatory — if no auth within 2 seconds,
the DC is dropped.

This is simpler because it doesn't require plumbing role through the SDP/peer
connection layer. The worker just stores one session-level host_token and
checks each DC's auth message against it.

```rust
// Per-DC state
let role: Arc<Mutex<Option<PeerRole>>> = Arc::new(Mutex::new(None));

// In on_message:
if let Ok(cmd) = serde_json::from_str::<Value>(&text) {
    if cmd.get("cmd") == Some(&json!("auth")) {
        let token = cmd.get("host_token").and_then(|v| v.as_str());
        let session_token = app_state.host_token.lock().unwrap();
        let is_host = token == session_token.as_deref();
        *role.lock().unwrap() = Some(if is_host { PeerRole::Host } else { PeerRole::Viewer });
        if !is_host {
            tracing::warn!("[DC] viewer authenticated (non-matching host token)");
        }
        return;
    }
    // All subsequent messages check role before dispatching
    let current_role = role.lock().unwrap();
    match current_role.as_ref() {
        None => {
            tracing::warn!("[DC] message before auth — dropping");
            return;
        }
        Some(PeerRole::Viewer) => {
            // Only binary input + ping allowed
            // (binary input is handled before JSON parsing)
            if !matches!(cmd_type, "ping") {
                tracing::warn!("[DC] viewer attempted privileged command: {}", cmd_type);
                return;
            }
        }
        Some(PeerRole::Host) => {
            // Full permissions
        }
    }
}
```

**Acceptance:** DC without auth is dropped after timeout. Viewer DC rejects
save/load/bitrate. Host DC has full access.

### Task 4: Enforce permissions in message handler

**Files:** `gv-worker/src/main.rs`

Audit every command path in the DataChannel message handler and add role
gating. Binary input (RetroArch format) is always allowed — the port field
identifies the player seat, not the peer role.

```rust
// Binary path (always allowed):
if msg.data.len() == 3 {
    // ... existing SetInput handling ...
    return;
}

// JSON path (role-gated):
let cmd: Value = match serde_json::from_str(text) {
    Ok(v) => v,
    Err(_) => { tracing::debug!("[DC] non-JSON, non-binary — dropping"); return; }
};

let cmd_type = cmd.get("cmd").and_then(|v| v.as_str()).unwrap_or("");

// Always allowed (no role check):
match cmd_type {
    "ping" => { /* respond with pong */ return; }
    "auth" => { /* handled above in role assignment */ return; }
    _ => {}
}

// Host-only commands:
let role = role_guard!();
match cmd_type {
    "save_state" => { /* existing handler */ }
    "load_state" => { /* existing handler */ }
    "set_bitrate" => { /* existing handler */ }
    "set_pattern" => { /* existing handler */ }
    "force_keyframe" => { /* existing handler */ }
    _ => tracing::debug!("[DC] unknown command: {}", cmd_type),
}
```

**Acceptance:**
- Host saves state → saved ✓
- Viewer saves state → dropped, warning logged
- Viewer sends binary input → processed normally
- Viewer sends ping → pong returned

### Task 5: Auth timeout — drop unauthenticated channels

**Files:** `gv-worker/src/main.rs`

If a DataChannel opens but never sends an `auth` message within 5 seconds,
close it. Prevents idle channels from lingering.

```rust
let dc_close = dc.clone();
let role_timeout = role.clone();
tokio::spawn(async move {
    tokio::time::sleep(Duration::from_secs(5)).await;
    if role_timeout.lock().unwrap().is_none() {
        tracing::warn!("[DC] auth timeout — closing channel");
        dc_close.close().await.ok();
    }
});
```

**Acceptance:** Open DataChannel without auth → closed after 5 s. Auth within
timeout → channel stays open.

### Task 7: Pass host_token from gv-server to worker at spawn time

**Files:** `gv-server/src/main.rs`

When gv-server processes a `start_game` command, extract `host_token` from
the payload and pass it to the worker via `--host-token` CLI arg or
`GV_HOST_TOKEN` env var.

```rust
let host_token = cmd.payload.get("host_token")
    .and_then(|v| v.as_str())
    .unwrap_or("");

let mut worker_cmd = Command::new(&worker_bin);
worker_cmd.env("GV_HOST_TOKEN", host_token);
// ... spawn ...
```

This ensures the worker knows the host token even before the first SDP offer
arrives (a belt-and-suspenders measure — the SDP offer from the host will also
carry it, but the env var establishes it immediately at worker startup).

**Acceptance:** Worker starts with GV_HOST_TOKEN set. Token matches the one in
the browser's start_game payload.

### Task 8: Wire host_token through gv-web → gv-server relay

**Files:** `gv-web/app/api/server/command/route.ts`

Ensure the command route passes through `host_token` in the `sdp_offer`
payload (it should already — it stores the entire payload as JSON). Verify
no field filtering is dropping it.

```bash
# Verify: POST sdp_offer with host_token, check gv-server receives it
curl -X POST /api/server/command -d '{
  "server_id": "...",
  "type": "sdp_offer",
  "payload": {"game_id": "2048", "sdp": "...", "host_token": "test-token"}

```

**Acceptance:** host_token survives the gv-web → gv-server → worker path
without being stripped.

### Task 9: Viewer input seat assignment

Currently all input goes to `port: 0` (player 1). When multiple viewers
connect, they should get distinct seats. This is out of scope for the
host token ADR but worth noting:

- Host always gets seat 0 (player 1)
- Viewers get assigned sequentially (seat 1, 2, ...)
- Seat assignment happens at connect time, survives reconnects
- Binary input's `port` byte is the seat number

This will be a follow-up ADR. For now, the host token ADR ensures
only the host can issue privileged commands.

---

## Non-goals

- Seat assignment / multi-seat input (separate ADR)
- Controller remapping per-player
- Chat or spectator features
- Persistent host identity across sessions (host token is per-session)

## Open questions

1. **Token format:** `crypto.randomUUID()` (browser) vs shorter random string.
   UUID is overkill but requires zero server-side generation. Fine for now.

2. **Auth message vs SDP-level tagging:** Auth message is simpler but adds
   one round-trip. SDP-level requires plumbing role through the peer connection
   layer. Auth message wins for simplicity.

3. **Viewer reconnect:** If the host disconnects and reconnects, should they
   re-authenticate? Yes — the same host_token is sent on every reconnect.
